# 计划：创建 /backlog skill

## 交付物一览

1. `skills/backlog/SKILL.md` — 新 skill 定义
2. `docs/BACKLOG.md` — 把现有内容替换为「新骨架 + 原有条目」，作为本轮 skill 的第一个示范/验证
3. `docs/7-创建backlog-skill/SUMMARY.md` — 收尾阶段产出（`/finish` 阶段写，不在本 PLAN 展开）
4. 无需改动 `install.sh`（逐目录扫描 `skills/*/` 已天然支持新增 skill）

## 设计决策

### 1. skill 的定位：追加 + 扩写 + 归类（交互式）

参照用户原话"只添加，删除在 finish 里做"——添加范畴内，skill 要主动帮忙，不做纯搬运：

- **不支持** 删除、修改、重排、优先级调整（删除归 `/finish`）
- **主动扩写**：拿到参数后按骨架模板（动机 / 目标状态 / 候选方向 / 风险 / scope）分析、补齐、给结构化草稿，跟人类讨论确认后再落盘（细节见第 3 节）
- **主动归类**：读出文件已有的 `##` 分类，给出归属建议，人类确认后再插入（细节见第 4 节）
- 删除职责已经在 `skills/finish/SKILL.md` step 2 里明确，不重复

### 2. 骨架模板：抽象自 whisper-input BACKLOG.md 开头

提炼出的通用骨架（新建文件时写入，已存在不动）：

```markdown
# Backlog

待启动的开发项清单。**本文件是权威来源**，取代各轮 `docs/N-*/SUMMARY.md` 里「后续 TODO」段的跨轮追踪职责 —— 那些段落继续保留，但只记录当轮发现的新线索；发现的新想法要立刻同步到这里。

**工作流**：

- **开新轮**时从下面的条目里挑一个作为 `docs/N-*/PROMPT.md` 的起点
- **收尾一轮**时从本文件**删掉**已完成的条目（不是打勾，是整条删，避免腐烂）
- **发现新想法**时立刻加进来，哪怕只写一行占位，之后再补完整

条目没有固定优先级 —— 选哪个做下一个看当时的心情和痛点。每条都尽量写成"未来自己或后续 agent 读完能接得住"的格式：**动机 / 目标状态 / 候选方向 / 风险 / scope**。

---
```

相对 whisper-input 原版的改动：

- 标题从 `# Whisper Input — Backlog` 改为通用的 `# Backlog`
- 删掉手工维护的「## 目录」段（每次追加都要同步目录会把 skill 职责无限扩大，放弃自动目录）
- 保留末尾的 `---`，作为「骨架段 / 条目段」的天然分隔

### 3. 扩写策略：模板化草稿 → 人类确认 → 落盘

拿到用户参数后**不立即落盘**，先在对话里输出一份基于骨架模板的扩写草稿：

1. **解析原始输入**：识别用户已经写了什么（是一句话？还是已有半结构化内容？）
2. **按模板五字段扩写**：
   - **动机**：基于原始输入推断"为什么要做"；信息不够就明确标注 `待补充`，而不是编造
   - **目标状态**：希望达到的效果、硬约束
   - **候选方向**：可能的实现路径（至少给 1-2 个候选，标注"未验证需要 spike"）
   - **风险 / 注意点**：可预见的坑、耦合面、UX 冲突
   - **scope**：粗估规模（小/中/大）+ 前置是否需要 spike
3. **给出标题建议**：用一个简明的三级标题 `### xxx`
4. **停下来等人类确认**：展示草稿，让用户可以：直接通过 / 调整某个字段 / 标"这条先不加 scope" / 改标题
5. 确认通过后，再写入文件

**关键边界**：

- 不强行塞满五个字段。如果原始输入明确说"只写一行占位"（对应骨架里"哪怕只写一行占位，之后再补完整"），skill 应尊重，直接落一行——扩写是**默认行为**，但人类可以用"只写一行"这类信号豁免。
- 扩写要**克制**：信息不够时宁可写 `待补充` 也不要脑补。模板价值在"提示缺哪些东西"，不是"帮 AI 造内容"。
- 扩写用中文，与现有文档风格一致。

### 4. 归类策略：读现有分类 → 推荐 → 人类确认 → 插入

1. **读文件**：如果 `docs/BACKLOG.md` 已存在，解析出所有 `## ` 级别的分类标题（跳过「已完成 / 不再追踪」这类 meta section）
2. **给出归属建议**：基于扩写草稿的内容，推荐归入某个已有分类，或建议新建一个分类（给出建议的分类名）
3. **展示建议**：跟扩写草稿一并展示给人类："建议放到 `## XXX` 下" / "建议新建 `## YYY` 分类"
4. **人类确认后插入**：
   - 归入已有分类：把新条目追加到该 `##` section 的末尾（下一个 `##` 之前，或文件末尾）
   - 新建分类：在文件末尾（或用户指定位置）追加 `## YYY` + 条目
5. **特殊情况**：
   - 文件不存在 / 骨架刚建时：没有已有分类，直接追问人类"给这条定个分类名（比如'识别能力'、'开发者体验'）"，或接受人类"先不分类"的选择——后者就直接裸挂在骨架下面
   - 条目之间用 `\n\n---\n\n` 分隔（与骨架末尾的 `---` 一致）

### 5. skill frontmatter

参照 `start` / `finish`（都 `disable-model-invocation: true`）—— backlog 添加条目是**人类有意识的动作**，不希望模型在别的流程里自主触发。

```yaml
---
name: backlog
description: 向 docs/BACKLOG.md 追加一个 backlog 条目；文件不存在时用通用骨架初始化
disable-model-invocation: true
---
```

### 6. 参数校验

- **必填**：参数是条目内容主体。若调用时无参数，skill 应追问"本次要加的 backlog 条目是什么"而不是静默退出。
- **不校验**内容格式、长度或关键词。

### 7. 顺手整理现有 docs/BACKLOG.md

现有 `docs/BACKLOG.md` 的开头（"待启动的开发项登记簿..."那段）是第 6 轮为了收尾需要临时写的，没有骨架的「本文件是权威来源」「工作流」「条目没有固定优先级」等结构。本轮顺手把开头**替换**为新骨架，但保留下面那条「系统整理 Claude Code Permission 配置」的完整内容。

这样做的价值：

- 验证骨架和现有条目能良好共存
- 给未来 `/backlog` 的使用者一个「正确长相」的参考样本
- 避免开发完 skill 后，本项目的 BACKLOG 反而是最不符合新规范的反面教材

## 实施步骤

### 步骤 1：创建 `skills/backlog/SKILL.md`

内容结构：

1. frontmatter（name / description / disable-model-invocation）
2. 触发说明：用户调用此 skill 表示要向当前项目的 BACKLOG 追加一个条目
3. 参数处理：
   - 有参数：进入步骤 4
   - 无参数：追问"本次要加的 backlog 条目内容是什么"，拿到后进入步骤 4
4. **扩写阶段**（第 3 节）：按五字段模板给出结构化草稿，标注 `待补充` 的字段；若用户说"只写一行占位"则跳过
5. **归类阶段**（第 4 节）：读 `docs/BACKLOG.md` 的 `##` 分类列表，给出归属建议（已有分类 or 新建）
6. **确认阶段**：把草稿 + 归类建议一起展示给人类，等确认或调整
7. **落盘阶段**：
   - 文件不存在 → 先 `mkdir -p docs`，写入「骨架 + 分类 + 条目」
   - 文件存在 → 插入到确认后的分类下（或新建分类后插入）
   - 条目间用 `\n\n---\n\n` 分隔
8. 完成后简短反馈：打印刚追加的条目标题行做 echo-back
9. **不调用** `/commit`，让用户自己决定是否立刻提交（符合"轻量追加"定位）

### 步骤 2：重写 `docs/BACKLOG.md`

把当前内容替换为：

- 新骨架（见上）
- `---` 后接现有的「## 系统整理 Claude Code Permission 配置」整条（内容不动）

### 步骤 3：不必跑 `install.sh`

只有在要立即使用新 skill 时才需要重跑 `install.sh`（项目 CLAUDE.md 已有此说明）。PLAN 不强制在本轮执行——留给用户自己决定何时部署。但在 SUMMARY 里会提醒。

### 步骤 4：`/finish` 阶段产出 SUMMARY.md + 更新 DEVTREE + commit

走既有 skill 链条，不在本 PLAN 展开。

## 风险与边界

- **骨架 vs 已有自定义开头**：本 skill 只在"文件不存在"时写骨架；如果某项目已手写了别的开头，skill 不会覆盖。这意味着不同项目的 BACKLOG 开头可能长不一样——可接受，本 skill 不做同步职责。
- **并发追加**：两次快速连续调用理论上可能出现竞态（罕见且损坏面小，失败重试即可），本版不处理。
- **参数里的换行**：CLI args 传 `\n` 的体验因 shell / skill 框架而异。扩写阶段会把用户参数当"原始输入"解析，最终落盘的是扩写后的结构化内容，对原始参数换行不敏感。
- **扩写 vs 占位**：扩写是默认行为，可能放大 skill 单次调用的对话长度。用户说"只写一行占位"时要尊重。
- **跨项目通用性**：骨架默认把"`docs/N-*/PROMPT.md` 起点""`docs/N-*/SUMMARY.md` 后续 TODO"这些跟全局 CLAUDE.md 约定的四步流程绑死。这对遵循本套方法论的项目是加分，对不遵循的项目没意义——可接受，毕竟 skill 就是给这套方法论配套用的。

## 不做的事

- 不做 `/backlog list`（查看条目直接读 Markdown 就好）
- 不做 `/backlog remove` / `/backlog edit`（删除归 `/finish`，编辑让用户自己改文件）
- 不自动维护 `## 目录`
- 不同步到跨项目的全局聚合视图
- 不做"脑补内容"的扩写——信息不够就写 `待补充`
