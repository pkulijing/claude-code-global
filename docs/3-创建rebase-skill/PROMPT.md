# 需求：创建 rebase skill

## 背景

在多开发项并行时（例如同一仓库的 `master` 和 `feat/xxx` 分支分别推进），两个分支容易发生分叉，需要通过 rebase 把一个分支的提交"接"到另一个分支上。

rebase 属于**重写历史**的操作，一旦处理不当容易导致：

- 覆盖掉队友在远程的新 commit
- 工作区未提交改动丢失
- 冲突解决到一半放弃后留下中间状态
- 在 worktree 场景下跑错目录

第一次遇到这种情况时容易手忙脚乱，因此希望沉淀成 skill，让 CC 每次都能按稳定流程处理。

## 需求

创建一个 `rebase` skill，功能是**诊断当前分叉状态 + 按清单引导完成 rebase**，而不是一键自动执行。核心要求：

1. **诊断优先**：skill 被调用后，先读出当前状态（当前分支、目标 base、分叉 commit 数、工作区是否干净、是否在 worktree 内），再给出针对当前情况的建议。
2. **安全网固化**：把容易漏的动作变成强制步骤：
    - rebase 前必须打备份 tag
    - 工作区必须干净（未提交改动需先 commit 或 stash）
    - push 时用 `--force-with-lease`，禁用 `--force`
3. **分段确认**：前置检查 → 执行 rebase → 冲突处理 → 推送验证，每段让人类确认再进下一步，不能一口气跑完。
4. **worktree 友好**：如果检测到在 worktree 中，明确提示当前目录属于哪个 worktree，避免在错的目录里 rebase。
