# 开发总结：创建 /devtree skill

## 开发项背景

希望解决的问题：随着项目开发历史变长，`docs/` 下的线性编号文件夹无法表达开发项之间的逻辑关系（功能迭代、平行分支、bug 修复等）。需要一个图形化的宏观文档，把所有开发项的分类与父子关系可视化为树状图，并在每次完成开发项时自动维护。

## 实现方案

### 关键设计

- **分类体系**：6 类（初建/功能/修复/重构/工程/探索），按「改动了什么」+「目的是什么」两个维度划分，重构和工程的关键区别在于前者改源代码结构，后者改构建/分发链路
- **树结构**：每个节点只有一个父节点，不允许多父。父节点选「最直接的上游」，不按时间顺序强行构造——并行分支追溯到共同的真实前序节点
- **两种模式**：首次运行时扫描所有 SUMMARY.md 从头重建完整树；已有 DEVTREE.md 时仅增量追加当前节点
- **可视化格式**：Mermaid `graph TD`，用 classDef 按类型配色，配套节点索引表

### 开发内容概括

- 新建 `skills/devtree/SKILL.md`：完整定义分类体系、父节点原则、重建/增量两种执行流程、DEVTREE.md 格式规范（含 Mermaid 模板）
- 修改 `skills/finish/SKILL.md`：在步骤 3 插入「调用 /devtree 更新开发树」，步骤 4 commit 时一并提交 DEVTREE.md

### 额外产物

无

## 局限性

- `/devtree` 的父节点判断依赖 LLM 语义分析，结果质量取决于 SUMMARY.md 的描述完整性；对描述模糊的开发项可能需要人工订正
- 增量模式下 LLM 需要读取完整的现有 DEVTREE.md，当树很大时上下文消耗较多

## 后续 TODO

- 考虑为 DEVTREE.md 增加「手动覆盖」机制：在节点索引表中允许人工标注父节点，增量更新时优先尊重手动标注
- 当树足够大后，可探索按功能域分组展示（Mermaid subgraph）
